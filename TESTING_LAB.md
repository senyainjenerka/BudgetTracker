# Лабораторная работа: Unit-тестирование и отладка React-приложения BudgetTracker

## 1. Цель работы

Цель лабораторной работы - настроить окружение для unit-тестирования React-приложения, написать тесты для бизнес-логики и компонентов, изучить отладку в DevTools и провести анализ производительности.

В проекте `BudgetTracker` для варианта 15 реализованы тесты для:

- `calculateBalance`
- `categorizeExpense`
- `TransactionItem`
- `BudgetChart`
- `ReportGenerator`

Дополнительно протестированы:

- `isWithinDateRange`
- `groupTransactionsForChart`
- `generateReport`

## 2. Теория

### 2.1. Что такое unit-тест

Unit-тест проверяет маленькую изолированную часть программы:

- одну чистую функцию;
- один React-компонент;
- один сценарий поведения.

Главная идея: если маленькие части приложения работают предсказуемо, всё приложение проще поддерживать, развивать и отлаживать.

### 2.2. Почему сначала тестируют pure functions

Чистая функция:

- получает входные данные;
- не зависит от внешнего состояния;
- не изменяет внешний мир;
- для одинаковых аргументов всегда возвращает одинаковый результат.

Такие функции проще всего тестировать, потому что не нужно поднимать браузер, DOM, роутер или состояние React.

### 2.3. Как работает Jest

`Jest` - это test runner. Он:

- находит файлы тестов;
- запускает их;
- сравнивает фактический результат с ожидаемым через `expect(...)`;
- показывает, какие тесты прошли и какие упали;
- умеет считать покрытие кода.

Основные конструкции:

- `describe()` - группирует тесты;
- `test()` - описывает один тестовый сценарий;
- `expect()` - проверяет результат;
- `jest.fn()` - создаёт mock-функцию для проверки вызовов.

### 2.4. Как работает React Testing Library

`@testing-library/react` тестирует компонент так, как его видит пользователь:

- компонент рендерится в виртуальный DOM;
- тест ищет текст, кнопки, чекбоксы, поля ввода;
- затем проверяет отображение и поведение.

Основные инструменты:

- `render()` - рендер компонента;
- `screen.getByText()` - поиск текста;
- `screen.getByRole()` - поиск элемента по роли;
- `fireEvent` - имитация действий пользователя.

### 2.5. Что такое jsdom

`jsdom` - это браузероподобная среда внутри Node.js. Благодаря ей Jest может тестировать React-компоненты без запуска настоящего браузера.

### 2.6. Что такое coverage

Покрытие показывает, какая часть кода была реально выполнена во время тестов.

Основные метрики:

- `Statements` - покрытие инструкций;
- `Branches` - покрытие ветвлений `if`, тернарных операторов и т.п.;
- `Functions` - покрытие функций;
- `Lines` - покрытие строк.

Покрытие не заменяет качество тестов, но помогает увидеть непротестированные участки.

## 3. Что реализовано в проекте

### 3.1. Настроено окружение Jest

Добавлены:

- `jest`
- `jest-environment-jsdom`
- `@testing-library/react`
- `@testing-library/jest-dom`
- `babel-jest`
- `@babel/preset-env`
- `@babel/preset-react`
- `identity-obj-proxy`

Созданы конфигурации:

- `jest.config.cjs`
- `babel.config.cjs`
- `test/jest.setup.cjs`

### 3.2. Добавлены npm-скрипты

```json
"test": "jest",
"test:watch": "jest --watch",
"test:coverage": "jest --coverage"
```

### 3.3. Вынесена бизнес-логика в отдельный модуль

Создан файл `src/utils/budget.js`, где находятся функции:

- `calculateBalance()` - считает баланс;
- `categorizeExpense()` - возвращает человекочитаемую категорию расхода;
- `isWithinDateRange()` - проверяет фильтрацию по датам;
- `groupTransactionsForChart()` - группирует транзакции для графика;
- `generateReport()` - формирует отчёт по транзакциям.

Это важно, потому что логика отделена от интерфейса и её можно тестировать независимо от React.

### 3.4. Добавлены компоненты под вариант

- `BudgetChart.jsx` - компонент графика для тестирования;
- `ReportGenerator.jsx` - компонент генерации отчёта;
- `Chart.jsx` оставлен как совместимый реэкспорт;
- `TransactionItem.jsx` улучшен с точки зрения доступности, чтобы тестировать его корректнее.

## 4. Какие тесты написаны

### 4.1. Тесты для pure functions

Файл: `src/utils/budget.test.js`

Проверяется:

- корректный расчёт баланса;
- игнорирование неподдерживаемых типов транзакций;
- категоризация расходов;
- поведение при неизвестной категории;
- фильтрация по диапазону дат;
- группировка для недельного и месячного графика;
- генерация агрегированного отчёта;
- корректная работа на пустом массиве.

### 4.2. Тесты для `TransactionItem`

Файл: `src/components/features/TransactionItem.test.jsx`

Проверяется:

- отображение названия, заметки, даты и суммы;
- знак `-` у расхода;
- знак `+` у дохода;
- состояние чекбокса;
- вызов обработчика при выборе транзакции.

### 4.3. Тесты для `BudgetChart`

Файл: `src/components/features/BudgetChart.test.jsx`

Проверяется:

- пустое состояние при отсутствии данных;
- отображение подписей графика;
- отображение итогов income/spent;
- смена периода через `select`.

### 4.4. Тесты для `ReportGenerator`

Файл: `src/components/features/ReportGenerator.test.jsx`

Проверяется:

- пустой отчёт;
- вывод итоговых сумм;
- отображение ключевых транзакций;
- отображение разбивки по категориям.

## 8. Как работать с DevTools

### 8.1. Elements

Во вкладке `Elements` можно:

- увидеть DOM-структуру React-приложения;
- проверить, как отрисован `TransactionItem`, `BudgetChart`, `ReportGenerator`;
- менять CSS в реальном времени;
- смотреть итоговые классы CSS Modules.

### 8.2. Console

Вкладка `Console` нужна для быстрой проверки данных и отладки логики.

Полезные примеры:

```js
console.log(report)
console.table(report.spendingByCategory)
console.group('Budget report')
console.log('Balance:', report.balance)
console.log('Top expense:', report.topExpense)
console.groupEnd()
```

`console.table()` особенно удобен для анализа категорий расходов.

### 8.3. Sources

Во вкладке `Sources` можно:

- открыть `src/utils/budget.js`;
- поставить breakpoint в `calculateBalance()` или `generateReport()`;
- обновить страницу;
- пошагово пройти выполнение через `Step over` и `Step into`;
- смотреть значения переменных в блоке `Scope`.

Что удобно отлаживать:

- фильтрацию транзакций;
- пересчёт графика;
- генерацию отчёта;
- обработчики выбора чекбоксов.

### 8.4. Network

Во вкладке `Network` можно проверить:

- какие JS и CSS файлы загружаются;
- сколько времени занимает загрузка;
- есть ли лишние или тяжёлые ресурсы;
- статусы запросов.

Для Vite-приложения здесь хорошо видно:

- загрузку модулей;
- статику;
- изменения при hot reload в режиме разработки.

### 8.5. Performance

Во вкладке `Performance` можно:

- записать профиль работы страницы;
- посмотреть, какие действия вызывают перерисовки;
- оценить стоимость рендера графика и списка транзакций.

Для этого проекта полезно профилировать:

- смену месяца;
- применение фильтров;
- переключение периода графика;
- добавление новой транзакции.

### 8.6. React Developer Tools

Нужно установить расширение `React Developer Tools`.

Оно даёт вкладки:

- `Components` - просмотр дерева React-компонентов;
- `Profiler` - анализ перерисовок.

В этом проекте можно смотреть:

- props у `TransactionItem`;
- props у `BudgetChart`;
- props у `ReportGenerator`;
- изменения состояния после фильтрации и добавления транзакций.

## 9. Lighthouse

### 9.1. Что это такое

`Lighthouse` - это встроенный аудит веб-приложения в Chrome DevTools.

Он оценивает:

- `Performance`
- `Accessibility`
- `Best Practices`
- `SEO`

### 9.2. Как запустить

1. Открыть приложение в браузере.
2. Открыть DevTools.
3. Перейти во вкладку `Lighthouse`.
4. Выбрать категории:
   `Performance`, `Accessibility`, `Best Practices`, `SEO`.
5. Нажать `Analyze page load`.

### 9.3. На какие метрики смотреть

- `FCP` - когда появился первый контент;
- `LCP` - когда загрузился самый крупный видимый элемент;
- `CLS` - насколько "прыгает" интерфейс;
- `Speed Index` - насколько быстро страница визуально наполняется.

### 9.4. Что анализировать в BudgetTracker

- быстро ли появляется список транзакций;
- не тормозит ли рендер `BudgetChart`;
- нет ли лишних перерисовок после фильтрации;
- нет ли тяжёлых ресурсов и лишнего CSS/JS.

### 9.5. Возможные рекомендации по улучшению

- уменьшить размер изображений и SVG;
- убирать неиспользуемый код;
- минимизировать лишние рендеры;
- кэшировать статические ресурсы;
- следить за размером бандла.
